## Inner Solar System — Near-Earth Objects

An interactive 3D visualization (Three.js + TypeScript + Vite) that propagates every body from its orbital elements and renders the inner solar system with its ~42,000 near-Earth objects, verified working in the browser at 60 fps.

### Required base — all met
- **3D solar system** — Sun, 8 planets with geometrically-correct orbit lines, and all 42k asteroids, each positioned by solving Kepler's equation from its own epoch. (Apophis, Eros etc. resolve to their real orbits.)
- **Interactive time** — play/pause, ±1/±30 day steps, a logarithmic speed slider (1 day/s → 10 yr/s, both directions), an 1900–2075 scrubber, a date picker, and "Now".
- **Real data via fetch** — loaded at runtime from `./data/` (relative URLs). A build-time `scripts/preprocess.mjs` converts the read-only `data/` into compact runtime assets (a 1.3 MB Float32 element buffer for the asteroids, etc.) copied into `dist/data/`. No fabricated positions, no network calls.
- **Usable & intentional** — orbit/zoom camera, starfield, legends, labels, tooltips, glassy UI.

### Open-ended directions implemented
- **Filter & search** — name/designation, PHA flag, Sentry list, orbit class, max MOID, min diameter.
- **Detail view** — full orbital + physical properties, **Sentry** impact-risk (joined by designation), and **Earth close-approach** history in lunar distances (the real Apophis 2029 0.1-LD pass shows up and is flagged).
- **Highlighting & encoding** — color asteroids by orbit class / PHA / Sentry risk / size, each with a legend; sizes scale with diameter.
- **Comets overlay** — toggleable, including hyperbolic/parabolic orbits (Barker / hyperbolic Kepler).
- **Focus & follow** — frame and lock the camera onto a selected body as time advances.
- **Shareable deep links** — URL hash encodes time, selection, and camera (verified to restore on reload).

### Key technical decisions
- **GPU Kepler propagation** for the 42k asteroids (orbital elements as vertex attributes; Newton iteration in the vertex shader) → zero per-frame CPU cost.
- **GPU picking** reuses the same shader to write id-colors into a 1×1 readback, so clicking shader-positioned points works precisely.
- Planets and comets propagate on the CPU (comets need elliptic/hyperbolic/parabolic handling).

### Hard rules
`data/` untouched; no external/runtime network calls; **clean `pnpm install && pnpm build` succeeds with no manual steps** (esbuild build script auto-approved via `pnpm.onlyBuiltDependencies`) and emits a self-contained `dist/` with relative asset/data URLs. Assumptions are documented in `README.md`.

One note: at an extreme zoom right into the Sun, fill-rate from overlapping point sprites drops the framerate; normal exploration views run 32–60 fps, and I capped max point size to mitigate the worst case.
